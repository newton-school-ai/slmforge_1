# Engine API Contract (v0)

This document defines the API contract shared between the CLI, the Web UI, and the engine. All three components consume the same request and response shapes. Any change to this contract must be reviewed by the maintainer and updated in the same pull request.

## Build Request

The CLI and Web UI submit build requests using the following JSON structure.

```json
{
  "sources": [
    {
      "type": "local",
      "path": "/data/train.jsonl"
    },
    {
      "type": "public",
      "id": "cnn_dailymail"
    },
    {
      "type": "synthetic",
      "generator": "feedback_summariser",
      "size": 20000,
      "seed": 42
    }
  ],
  "task_type": "summarisation",
  "base_model": "auto",
  "lora": {
    "r": 16,
    "alpha": 32,
    "dropout": 0.05
  },
  "training": {
    "epochs": 2,
    "batch_size": 16,
    "lr": 0.0002
  },
  "eval": {
    "llm_judge": false
  }
}
```

### Source Types

#### `local`

A dataset stored on the local filesystem.

**Example**

```text
/data/train.jsonl
```

#### `public`

A publicly available dataset identifier.

**Example**

```text
cnn_dailymail
```

#### `synthetic`

A dataset generated by one of the project's synthetic generators.

**Example**

```text
feedback_summariser
```

#### `internal`

Reserved for internal builds.

This source type is intentionally stubbed in the public repository.

### Task Types

Supported values:

* `classification`
* `summarisation`
* `qa`
* `instruction`
* `chat`
* `auto`

### Base Model

Supported values:

* `auto`
* Any valid Hugging Face model identifier.

**Example**

```text
microsoft/Phi-3-mini-4k-instruct
```

## REST API Endpoints

### `POST /builds`

Creates a new build.

#### Request

The request body must follow the **Build Request** schema defined above.

#### Response

```json
{
  "build_id": "build_2026_06_10_142231",
  "status": "queued"
}
```

### `GET /builds`

Returns all known builds.

### `GET /builds/{build_id}`

Returns the current state of a specific build.

### `WebSocket /builds/{build_id}/stream`

Streams build progress events until completion.

## Streamed Build Events

The engine emits newline-delimited JSON events through the WebSocket connection.

```jsonl
{"event":"discover_sources","files":[...]}
{"event":"detect_task","task_type":"summarisation"}
{"event":"plan","vram_gb":18,"est_minutes":24}
{"event":"epoch","epoch":1,"train_loss":1.23,"val_loss":1.05}
{"event":"checkpoint","path":"builds/build_.../checkpoint-epoch-1"}
{"event":"eval","metrics":{"rouge_l":0.62,"accuracy":0.91}}
{"event":"complete","build_id":"build_2026_06_10_142231","usage_url":"builds/build_.../USAGE.md"}
```

### Event Definitions

| Event              | Description                                       |
| ------------------ | ------------------------------------------------- |
| `discover_sources` | Source discovery completed                        |
| `detect_task`      | Engine detected the dataset task type             |
| `plan`             | Hardware requirements and training plan generated |
| `epoch`            | Training progress update                          |
| `checkpoint`       | Model checkpoint saved                            |
| `eval`             | Evaluation metrics generated                      |
| `complete`         | Build completed successfully                      |

## On-Disk Build Layout

```text
builds/<build_id>/
  adapter/
  dataset_card.md
  model_card.md
  eval_report.md
  USAGE.md
  serve.sh
```

### Directory Contents

| File              | Description                                         |
| ----------------- | --------------------------------------------------- |
| `adapter/`        | LoRA adapter weights                                |
| `dataset_card.md` | Dataset metadata, sources, splits, and licence      |
| `model_card.md`   | Base model, hyperparameters, and evaluation summary |
| `eval_report.md`  | Complete evaluation report                          |
| `USAGE.md`        | Example inference commands and integration notes    |
| `serve.sh`        | Helper script for serving the trained model         |

---

## Versioning

The engine contract is versioned.

Breaking changes include:

* Request schema changes
* Response schema changes
* Streamed event changes
* On-disk layout changes

Breaking changes require:

1. Bumping the contract version (for example, `v0` → `v1`).
2. Updating this document in the same pull request.
3. Keeping the CLI, Web UI, and engine synchronized with the same contract version.
4. Obtaining maintainer review before merge.
